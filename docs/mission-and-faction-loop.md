# Synapse mission and faction loop

This note turns the high-level direction into a concrete playable loop.

It exists so mission design, hub UX, faction standing, and travel all support
the same game instead of drifting apart.

## Core loop

The canonical near-term loop is:

1. enter a hub or safe station
2. review available faction contracts
3. stack compatible missions headed toward the same route when possible
4. depart and travel to the target system or local zone
5. resolve the local objective
6. survive escalation, elite encounters, or boss pressure
7. return, extract, or hand off for payout, standing, and upgrades

This is a mission-driven action loop, not a sandbox where the player is meant to
invent all purpose alone.

## Faction role categories

The first working faction structure should assume three broad categories:

### Authority

- patrols
- convoy protection
- bounty work
- suppression and response contracts

### Outlaws

- raids
- theft
- smuggling
- ambush and extraction jobs

### Rebels

- sabotage
- rescue
- intel recovery
- disruption of stronger powers

These are directional categories, not final lore names.

## Side selection philosophy

Synapse should let the player drift toward a side by repeatedly taking its work.

The first implementation should avoid a permanent one-time morality lock.

Instead, the game should favor:

- standing that rises with repeated support
- consequences in the form of better contracts, harder enemies, and different
  mission framing
- the feeling that the player is choosing who they work for, not filling a
  neutral task list

## Contract rules

Contracts should follow these rules:

- destination is clear before acceptance
- mission type is clear before acceptance
- payout and standing impact are readable before acceptance
- several contracts going to the same route should be stackable
- contracts should describe why the player is going there, not just what to kill

The player should be able to build a run instead of accepting a single isolated
errand.

## Objective families

The early objective set should stay close to the combat loop:

- bounty hunt
- intercept / ambush
- escort / defense
- scan / recover under pressure
- assault on a defended target
- elite duel or boss contract

Each objective should create a clean path into the combat pillars already
documented elsewhere:

- pressure
- openings
- punish windows
- stagger payoff

## Hub rules

The hub exists to reset pacing between missions.

It should provide:

- mission board / contract review
- debrief / payout
- standing changes
- lightweight upgrade decisions
- just enough faction flavor to give context to the next choice

The hub should not become a life-sim layer.

## Co-op compatibility rule

Synapse does not need multiplayer-first architecture yet, but the loop should be
co-op compatible later.

That means:

- contracts should be understandable and completable by a group as well as a
  solo player
- destinations and mission states should be shareable
- faction standing should be able to scale to shared play without redesigning
  the whole contract model

The right question is not “how do we make an MMO now?”

The right question is “does this solo mission loop still make sense if two or
three pilots share it later?”
